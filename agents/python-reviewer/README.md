# Python Reviewer 使用说明

`python-reviewer` 是一个面向 Python 代码变更的审查代理，重点关注安全、健壮性、类型注解、Pythonic 风格与性能问题，帮助团队在提交前尽早发现高风险缺陷。

## 功能定位

- 面向 Python 代码审查场景，优先检查已修改的 `.py` 文件。
- 审查重点覆盖：安全、错误处理、类型注解、代码风格与性能优化。
- 输出结构化审查结果，按严重级别分组，便于快速修复。

## 推荐使用时机

- 任意 Python 代码改动完成后。
- 提交 PR / 合并分支前的质量检查。
- 需要统一代码规范与提升可维护性时。

## 工作流程

调用该审查代理时，按以下步骤执行：

1. 运行 `git diff -- '*.py'` 查看 Python 文件变更。
2. 运行静态分析命令：
   - `ruff check .`
   - `ruff format --check .`
3. 聚焦修改过的 `.py` 文件进行审查。
4. 按优先级输出问题与修复建议。

## 审查优先级与关注点

### CRITICAL（安全）

- SQL 注入（如 f-string 拼接 SQL）
- 命令注入（未校验输入直接传入 shell）
- 路径穿越（用户可控路径未做规范化校验）
- `eval/exec` 滥用
- 不安全反序列化
- 硬编码密钥
- 弱加密（如 MD5/SHA1）
- YAML 不安全加载（unsafe load）

### CRITICAL（错误处理）

- 裸 `except`（例如 `except: pass`）
- 吞掉异常导致静默失败
- 缺少上下文管理器（应优先使用 `with`）

### HIGH（类型注解）

- 公共函数缺少类型注解
- 过度使用 `Any`
- 可空参数缺少 `Optional`

### HIGH（Pythonic 模式）

- 可用列表推导却使用 C 风格循环
- 使用 `type(x) ==` 而不是 `isinstance()`
- 魔法数字未抽象（可考虑 `Enum`）
- 循环中字符串拼接未使用 `"".join()`
- 可变默认参数（如 `def f(x=[])`）

### MEDIUM（性能）

- 大数据处理可改为生成器
- 成员检查应优先 `set`
- 循环中重复计算
- 手动初始化字典可用 `collections.defaultdict`

### LOW（风格）

- 遵循项目编码规范
- 行宽控制（80 字符）
- `snake_case` 命名
- import 分组顺序：标准库 -> 第三方库 -> 项目内部

## 输出格式约定

审查结果建议使用如下结构：

```text
## 审查结果

### CRITICAL
- [文件:行号] 问题描述 -> 修复建议

### HIGH
- [文件:行号] 问题描述 -> 修复建议

### MEDIUM
- [文件:行号] 问题描述 -> 修复建议

总结：X 个 CRITICAL，Y 个 HIGH，Z 个 MEDIUM
```

## 注意事项

- 优先给出可执行、可落地的修复建议。
- 对高风险问题（尤其安全类）应优先阻断并明确整改方式。
- 审查结论尽量包含具体文件与位置，便于快速定位。
